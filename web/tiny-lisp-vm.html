What we actually builtOne language, three engines, and a rule: measure, don't guess.

The language is deliberately small — integers, cons cells, lambdas and closures, if/let/define, and a handful of primitives. No strings, no division, no floats. That smallness was the point: it let us implement the same semantics three ways and compare the architectures honestly, instead of comparing who optimized their parser hardest.

Tree-walker

lisp.c

The textbook approach: recursively evaluate the cons-tree directly. Simple, obvious, and a perfectly good baseline.

Exhibit B

CEK machine

cek.c

Control / Environment / Kontinuation registers; every step a real tail call, continuations on the heap. Elegant — and it can do deep non-tail recursion the others can't.

› 2.2× slower · surprised us
Exhibit C · finalist

Bytecode VM

bytecode_gc.c

Compile the AST to a flat instruction array once, then run it on a stack machine. O(1) global access, tail-call opcode, and now a garbage collector.

› 2.3–3.9× faster · winner

The bytecode VM won across every benchmark — by the most (3.9×) on allocation-heavy list reversal, by the least (2.3×) on call-bound tak. The real shock was the CEK machine coming in slower than the humble tree-walker. The reason is humbling: clang already eliminates the tree-walker's self-recursive tail calls at −O2, so the feature the CEK machine was built to provide, the baseline got for free — while paying nothing for heap-allocated continuations.

01

The part where we were wrong, repeatedlyEvery confident performance prediction met a benchmark that disagreed.

This is the actual texture of low-level work, and the reason the whole thing is built on measurement rather than intuition. A running tally of hypotheses that felt obvious and turned out false:

⚠ Falsified by measurement
  1. Environment lookup is the hotspot. → It wasn't — variable lookups walk only ~2 cells on average. Optimizing it bought nothing.
  2. A switch-dispatch will speed up the tree-walker. → Worth ~5%, entirely eaten by the −Oz size build. Reverted.
  3. The CEK machine's proper tail calls will be a win. → The compiler already gave the tree-walker tail-recursion for free. Net loss.
  4. Tail-call optimization makes recursion unbounded. → It fixes the stack, not the heap. Long loops still exhausted the arena — until GC.
  5. GC overhead comes from collection (or register hoisting). → Collection is nearly free; hoisting costs ~6%. The real cost is something else entirely — see below.
02

The garbage collectorWhy mark-sweep, and what five pre-registered hypotheses actually showed.

We chose mark-sweep over a copying collector for two reasons specific to this machine. First, every heap object is a uniform cons cell, which erases mark-sweep's usual weakness — fragmentation simply can't happen when all cells are the same size. Second, it's non-moving: our pointers are array indices scattered across the operand stack, the environment, saved frames, globals, and even constants embedded in the bytecode. A moving collector would have to find and rewrite every one of them. A non-moving collector only has to mark them. We wrote the predictions down before testing.

H1 ✓

Unbounded tail recursion

With TCO flattening the stack and GC flattening the heap, a long loop should run in fixed memory forever. Confirmed, decisively.

countdown(10,000,000) ran to completion in a 512-cell arena — GC fired 118,577 times
H5 ✓

The root set is complete

The make-or-break test: shrink the heap until GC fires constantly mid-computation, then check correctness. A missing root would corrupt or leak. Held.

fib(18)=2584 correct while GC ran 120× mid-eval · full suite 19/19 · quoted data survives collection
H2 ⚡

Where the overhead really lives

The GC build runs ~1.3–1.7× slower than the no-GC build — but not because of collecting. Collection itself is nearly free (gc=0 and gc=4 runs tie). The cost is an optimization barrier: once cons can reach the collector, the compiler can no longer treat allocation as side-effect-free, and pessimizes the hot loop around every allocation.

same cons, slow path without gc() = 1.11× · with gc() = 1.66× — that gap IS the barrier
03

Anatomy of the finalistFrom source text to a collected stack machine.

01
Read
A recursive-descent reader turns text into a cons-tree. Quote shorthand, tolerant of a missing paren.
02
Compile
The tree is walked once into a flat u32 instruction stream — lexical addresses for locals, an O(1) slot for globals, a dedicated OP_TAILCALL for calls in tail position.
03
Run
A stack machine executes the stream. Tail calls reuse the current frame, so loops cost no stack. Frames and closures are ordinary cons cells.
04
Collect
When the arena fills, a mark stack traces every root, then a sweep threads dead cells onto a free list. Indices never move.
One value · 32 bits
payload 30 bits of value
tag 00 fixnum — a 30-bit integer
tag 01 cons — index into the arena
tag 10 symbol — interned, stable
tag 11 special — nil, true, error…
The low two bits carry the type. Because the collector only manages cons cells — symbols live in a separate stable table — tracing is a tight loop over two fields per cell.

It is, proudly, a toy.

Fixnums wrap at 30 bits. There are no strings, no floats, no division. The benchmark numbers are V8-on-wasm and a standalone runtime would differ. None of that is the point. The point was to build the same idea three ways, hold every belief about it up to a measurement, and keep only what survived. What survived is a 9 KB, zero-import bytecode machine with a collector you can starve to 512 cells and still get the right answer.
