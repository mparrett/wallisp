# Repo audit — 2026-07-21

Full-repo audit for correctness, maintenance, performance, and public-release
readiness. Run as four parallel reviews (C engines, JS harness/tests/web, repo
hygiene, docs/build consistency). All test suites were green at audit time:
parity 136/136 × 8 engines, test_bc 70/70, test_session 35/35, parity_strings
44/44, parity_callcc 16/16, standalone 80/80, reader_sugar 21/21, futamura
regressions pass.

Findings are grouped by theme, most actionable first. Line numbers are as of the
audit commit (`f608558`).

---

## 1. Memory safety — the real release exposure (C engines)

The engines are internally consistent and the `_gc` rooting discipline is
careful. The exposure is at the two untrusted boundaries (reader, printer) plus
a couple of pure-Lisp-reachable corruption paths. None of these are the
expected "arena exhaustion returns `<error>`" case — they are corruption / crash
/ hang paths a fuzzer would find quickly.

### HIGH — forgeable `%string` wrapper → OOB read/write (`bytecode_gc.c`)
`is_string()` (`:154`) checks only that a cons's car is the interned `%string`
symbol; the heap offset in the cdr is an unvalidated fixnum. Pure Lisp can forge
one: `(cons '%string 999999999)`. Then `str_off`/`str_len` (`:155-156`, no
bounds check) read a length + that many bytes at `strheap + off` — wild OOB read
(trap, or leaks unrelated memory into output via `print_val`/`display`). During
GC, `mark()` (`~:530`) does `*(u32*)(strheap+off) |= 0x80000000` guarded only by
`off < STR_HEAP_BYTES`, so `off = STR_HEAP_BYTES-1` writes 3 bytes past
`strheap[]`. No arena pressure needed. **Standout bug.** Fix: validate offsets,
or use an unforgeable tag rather than a plain symbol + fixnum.

### HIGH — unbounded shadow stack in `lisp_gc.c`
`SAVE_MAX` is 16384 and nothing checks `R_ssp < SAVE_MAX` before
`R_save[R_ssp++]` (`:96`, `:302-303`, and every push in `eval`/`eval_list`).
`eval_list` pushes ~3 slots per level and is reachable with a *flat* arg list
(reader builds lists iteratively, so paren nesting is no protection):
`(f a a a … a)` with ~6000 args overruns into the adjacent `markbit`/`markstk`
GC arrays — silent GC-metadata corruption (can free/mismark live cells), not a
clean `<error>`.

### HIGH — unchecked operand stack in the bytecode VMs
`csp` is checked against `CALL_MAX` but `vsp` is never checked against
`VSTACK_MAX` (65536): `bytecode.c:302-304,314`, `bytecode_gc.c:514-516,590`,
`standalone/wallisp.c:387-389,437`. A flat call/list with >65k args (parseable
without deep reader recursion) pushes operands past `vstack[]` into the adjacent
`ret_ip`/`ret_env` arrays — silent corruption.

### MED — reader edges (shared `reader.h`, affects all engines)
- **Integer-literal overflow / silent fixnum wrap** (`reader.h:57`, inlined at
  `standalone/wallisp.c:143`): `val = val*10 + (c-'0')` on `i32` with no
  `fits_fix` check. `2000000000` silently wraps; `99999999999999999999` is
  straight signed-overflow UB. The arithmetic prims already clamp with
  `fits_fix` — the reader should too.
- **Unbounded reader recursion** (`reader.h:36-48,67-85`):
  `read_list`↔`read_expr`↔`read_string` recurse per nesting level, no depth cap.
  A few tens of thousands of `(((((…` overflow the C stack and trap the module
  (hard crash / DoS) instead of returning `<error>`.

### MED — cyclic-structure print hang (`print_val`, identical in every engine)
`lisp.c:405-411`: cdr spine walked with `while(is_cons(p))`, each car printed
recursively. Every engine ships `set-car!`/`set-cdr!`, so
`(define x (cons 1 2)) (set-cdr! x x) x` makes the last-result printer loop
forever; a car-cycle recurses to stack overflow. No visited-set/depth guard.

### LOW
- **16-char symbol truncation** (`intern`, e.g. `lisp.c:85-94`): identifiers
  sharing a 16-char prefix intern to the same symbol and alias each other's
  bindings, silently. Fixed `SYM_CHARS`, no diagnostic.
- **`strheap-reset` dangling wrappers** (`bytecode_gc.c:259-261`): rewinding
  `strheap_top` leaves later-allocated string values pointing into reclaimed
  region — bounds-guarded (no OOB) but a Lisp-level use-after-free. Documented as
  a caller contract.
- **Standalone reader drift** (`wallisp.c:120-126`): the inlined standalone
  reader omits `,` from `skipws`/`is_delim`; shared `reader.h` treats comma as
  whitespace. Divergent tokenization, same lineage.

**Clean:** `native/main.c`, `native/bench.c`, `baselines/bench.c` (inputs
clamped, bounds-checked). `cek.c`/`cek_gc.c` use heap-allocated continuations
(no C-stack growth), gate on `g_oom`, careful rooting — no provable missed root.

**Suggested pre-release fix set:** cap reader + printer depth; bound integer
literals to `fits_fix`; bounds-check `R_ssp` and `vsp`; validate/replace the
string wrapper. These close every corruption/crash/hang path above.

## 3. Docs vs code drift (numeric + capability rot)

Doc *architecture* is sound (README → ENGINES/DEV/FINDINGS, plan docs annotated
when closed, every runnable claim I exercised works). The rot is concentrated in
**hard numbers and capability claims embedded in prose**:

- **HIGH — `set-car!`/`set-cdr!` documented as bytecode_gc-only.** README (~:35)
  + DEV.md (:26-29) say other engines leave them unbound. Falsified:
  `lisp.c:381` binds both; FINDINGS "PR2c" ported mutation to all 8. Only
  *strings* remain bytecode_gc-exclusive.
- **HIGH — `set!` missing from DEV.md's special-forms list** (shipped to all 8 in
  PR2; examples/ and standalone/ READMEs list it — three specs now disagree).
- **HIGH — parity count "55 programs" in README:14 is now 136.**
- **MED — ISA lists 11 opcodes; enum has 14** (`bytecode_gc.c:375`):
  `OP_SETL`/`OP_SETG`/`OP_TAILCALL` missing from DEV.md.
- **MED — engine line counts stale in ENGINES.md** (claimed vs actual off by up
  to ~300 lines for bytecode_gc); README gives a third inconsistent set. Pick one
  home (ENGINES.md) or generate them.
- **MED — `-fno-builtin` documented as bytecode_gc-only** (DEV.md + CLAUDE.md);
  `build.sh:18` applies it to **all** engines (LLVM 20+ synthesizes
  `strlen`/`memset` otherwise, breaking zero-imports). A hand-build per DEV.md's
  flag list produces a broken module. **This CLAUDE.md build-gotcha is now
  wrong** and should be corrected. (`-mtail-call` for cek/cek_gc is accurate.)
- **MED — entire Futamura workstream undocumented at top level** (prototype/
  futamura/, docs/futamura.html, three harness scripts, 8 root `residual_*.wasm`,
  five FINDINGS H-sections). A visitor who greps `residual_*.wasm` finds no
  explanation outside a 2825-line FINDINGS.md. Surface it in README/DEV or label
  it internal.
- **MED — DEV.md file map is significantly incomplete** (omits parity.mjs and
  most harness scripts, reader.h, and whole dirs examples/ standalone/ baselines/
  tests/).
- **LOW** — engines/README says "all four" over a table of eight; examples/README
  cites `lisp-cli.mjs:14` (actually :17); DEV open-thread #3 marked DONE but still
  listed; h11 preregistration says "pending" but was measured + falsified.

**Verified correct:** `gap_closure_plan.md` and `shared_reader_plan.md` are
properly closed out (not stale). Every documented example command reproduces its
stated result.

## 4. Maintenance / ergonomics (JS harness, tests, web)

Scripts are in unusually good shape — no dead references, no bit-rotted engine
lists, checked-in HTML in sync with templates, web build scripts are defensive
and self-verifying. Release friction is ergonomic, not correctness:

- **HIGH — no unified test runner; `build.sh` runs no tests.** No
  package.json/justfile/Makefile/CI. 8 test entry points must each be found and
  run by hand — a contributor can `build.sh`, see green, and ship a parity break.
  Fix: a ~30-line `justfile`/`run_tests.sh` that runs all suites.
- **HIGH — `harness/bench.mjs:28-34` ENOENT on fresh clone.** Loads
  `lisp_big.wasm` etc., which are build artifacts (not tracked). README claims
  the harness works without building — false for bench. Emit a "run build.sh"
  hint, or guard.
- **MED — `tests/reader_sugar.sh:11-12` wrong bootstrap advice.** Defaults to
  `./native_cli_lisp` (untracked) and says "run ./build.sh first", but native
  CLIs need `build.sh --native`.
- **MED — reader sugar has zero wasm-engine coverage.** Sugar lives in shared
  `reader.h` (all 8 engines) but is tested only via the native CLI; parity.mjs
  has no sugar programs. A wasm-side sugar regression ships untested.
- **MED — macOS/Homebrew hardcoding in scripts the tests call.**
  `harness/disasm.sh:20` and `prototype/futamura/build.sh:31` hardcode Homebrew
  llvm paths (+ BSD `stat -f%z`); the latter is invoked by
  `test_futamura_regressions.mjs` and `bench_futamura.mjs`, so on Linux those
  tests fail with an obscure clang error instead of "install llvm". Root
  `build.sh` sets no PATH — three different conventions.
- **MED — wasm-instantiation boilerplate duplicated ~16×** and already diverged:
  `bench_futamura.mjs:103`, `bench_preproc.mjs:45`,
  `test_futamura_regressions.mjs:36` cache `memory.buffer` once — a stale-buffer
  bug the moment an engine grows memory (other drivers refetch per call;
  `game.mjs:75` even comments on it). One shared `harness/lib/vm.mjs` fixes it.
- **LOW** — `game.mjs` shows a bare `<error>` frame with no diagnosis when a game
  file fails to compile; `tiny-lisp-vm.html` discards `e.message` and treats any
  output containing the substring "error" as failure (`(quote error)`);
  `standalone/build.sh --check` prints success when node is missing;
  `web/build-standalone.sh` produces `web/bytecode_gc.wasm` shadowing the tracked
  root artifact with different flags; `disasm.mjs` silently truncates >8KB input;
  `web/game.html` needs the jsdelivr CDN despite a "opens from file://" comment.

## Recommended order of operations

**Must-do before public (small, high-leverage):**
1. Add LICENSE.
2. Remove the opus48 chat transcript.
3. Fix the memory-safety HIGHs (string wrapper, `R_ssp`/`vsp` bounds) + reader
   depth/integer-literal caps + printer cycle guard — the fuzzer surface.

   misleading anyone who hand-builds).
5. Fix the three HIGH doc drifts (mutation-is-universal, `set!` in the spec,
   parity count) and the bench.mjs fresh-clone ENOENT.
6. Add a `justfile`/`run_tests.sh` unified runner; add README prerequisites
   (Homebrew llvm/lld on macOS, a tested Node version).

**Nice-to-have:** shared `harness/lib/vm.mjs`; regenerate/consolidate engine
line counts into ENGINES.md; surface or label the Futamura workstream; add
wasm-side reader-sugar parity cases; relativize absolute paths in archived docs.

**Explicitly fine as-is:** `.gitignore` scheme, tracked-binary policy, git
history, plan-doc closeout, CEK rooting, web template/HTML sync.
