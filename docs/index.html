A measurement study

Eight tiny Lisps, one set of benchmarks

What you actually learn when you build the same language eight ways, pre-register predictions, and let the numbers refute you.

wallisp is a tiny Lisp implemented eight ways in freestanding C, compiled to WebAssembly with zero imports, and driven by measurement to a finalist: a bytecode VM with tail-call optimization and a hand-rolled mark-sweep GC. Along the way, four pre-registered hypotheses were tested and most of them got refuted — including some that looked obviously correct on paper.

The project isn't a Lisp tutorial. It's a study in how interpreter design choices interact with their compilation substrate, told through eight implementations of the same evaluator.

1What was built

Three evaluator architectures × three garbage-collection strategies, eight engines in the grid:

no GC mark-sweep region-drop
tree-walker lisp.c lisp_gc.c lisp_region.c
CEK machine cek.c cek_gc.c — not built
bytecode VM bytecode.c bytecode_gc.c — not built

The two empty cells are honest. Region-drop requires the live state to be a downward-only chain anchored at a stable bottom (the global env). Neither CEK (heap continuations are the live state, they churn mid-eval) nor bytecode (operand stack + call frames churn mid-eval) naturally satisfies that invariant. Region-drop is a tree-walker-shaped collector, not a universal one.

The tree-walker has a fourth variant too — lisp_trampoline.c, an explicit while(TRUE) form of the recursive eval — built to verify a measurement claim, not to test a new design point. We'll come back to it.

2The headline: bytecode wins, by a lot

Bytecode is 1.7–7× faster than every other engine on every benchmark in the suite. Here are all eight on a single program (fib 24), measured in Node/V8:

Why bytecode wins is the same on every benchmark: it attacks the axis V8's JIT doesn't already cover. Special-form recognition is compiled once into opcodes (no if(op==s_quote)...if(op==s_if)... cascade per call). Variables are addressed by lexical (depth, index) instead of walked via cons-cell association lists. The operand stack carries values between operations instead of cons'd argument lists allocating on every call. None of these are exotic tricks; they're the standard reasons a stack VM beats a tree-walker.

The bytecode hot loop, abbreviated:

case OP_LOADL: {                          // load local — O(1) lexical address
  u32 d = code[ip++], i = code[ip++], f = R_env;
  while (d--) f = cdr(f);
  u32 v = car(f); while (i--) v = cdr(v);
  vstack[R_vsp++] = car(v); break;
}
case OP_CALL: {
  u32 n = code[ip++]; u32 fn = vstack[R_vsp-n-1];
  if (is_prim(fn)) {
    u32 id = fn>>2;
    if (n == 2 && id >= PR_ADD && id <= PR_LT) {   // inline 2-arg arith — no arg-list cons
      u32 a = vstack[R_vsp-2], b = vstack[R_vsp-1], r;
      switch (id) {
        case PR_ADD: r = a+b; break;
        case PR_SUB: r = a-b; break;
        // ...
      }
      R_vsp -= 3; vstack[R_vsp++] = r;
    } else { /* general path */ }
  }
  // ...
}

The point isn't that this code is clever; it's that every operation does the minimum work and nothing more. A tree-walker's (+ 1 2) traverses three cons cells, dispatches through six if branches, builds a two-cell arg list, and calls into apply_prim. The bytecode (+ 1 2) is two OP_CONSTs and one OP_CALL that runs the inlined add directly on the operand stack. The constant factor adds up.

3The H1 surprise: clang already gave us tail calls

The original motivation for the CEK machine (engine #2 chronologically) was that the tree-walker would blow the C stack on deep tail recursion. A recursive eval() calling itself for the body of (if cond then else) stacks up a C frame per Lisp call. For (countdown 1000000), that's a million C frames. Crash.

So we built CEK: explicit Control/Environment/Kontinuation registers, every state transition a wasm tail call via __attribute__((musttail)), recursion depth living in heap-allocated continuations instead of the C stack. Beautiful machine. 2.2× slower than the tree-walker on fib.

Then we ran (countdown 1000000) on the tree-walker just to see how it failed. It returned done.

clang -O2 was already doing tail-call elimination. The recursive return eval(...) self-call gets transformed into a loop at compile time. The C stack stays flat. The motivation for CEK was a problem the compiler had already solved.

This is the kind of finding that looks obvious in retrospect and is invisible in advance. The mal project's Step 5 prescribes the canonical Lisp TCO pattern: wrap eval in while(TRUE), rebind ast and env on tail branches, continue instead of recursing. Our lisp.c doesn't have that loop. clang's TRE writes it for us.

To verify this framing isn't just a story, we built lisp_trampoline.c — the explicit mal step 5 form — and measured it against lisp.c:

lisp.c — recursive

if (op == s_if) {
  u32 t = eval(car(cdr(x)), env);
  if (t == ERR) return ERR;
  if (!is_nil(t))
    return eval(car(cdr(cdr(x))), env);
  else
    return eval(car(cdr(cdr(cdr(x)))), env);
}

lisp_trampoline.c — explicit loop

if (op == s_if) {
  u32 t = eval(car(cdr(ast)), env);
  if (t == ERR) return ERR;
  ast = is_nil(t)
    ? car(cdr(cdr(cdr(ast))))
    : car(cdr(cdr(ast)));
  continue;
}

The two engines:

  • run within 0.5% of each other on wasm (1.005× mean ratio, every individual benchmark within ±2%)
  • produce wasm modules two bytes apart (9840 vs 9838 bytes, a 0.02% difference)
  • both pass countdown(1,000,000) and mutual (even? 1000000)

The recursive form is paying no hidden cost. clang's TRE is doing exactly the transformation mal step 5 prescribes, and the modern C compiler hides nothing here. The "explicit trampoline is the canonical Lisp TCO pattern" stance is correct as a portability guarantee — works on any C compiler — but the recursive form is equally fast when the compiler has TRE.

4The GC tax: same collector, three different prices

The mark-sweep collector is the same code in all three GC engines — the same free list, the same mark bits, the same sweep. But the runtime tax it costs varies wildly. Comparing each GC engine to its no-GC sibling:

The three mark-sweep engines pay strikingly different prices, and the substrate matters: V8 amplifies the tax for some engines much more than for others. bytecode_gc pays almost nothing — the tax is a rounding error. cek_gc pays 1.83× on wasm, a 40% jump over its 1.28× native cost.

The mechanism: an optimization barrier per cons callsite

Adding the collector required splitting cons() into a fast bump path and an out-of-line slow path that handles free-list reuse and GC:

static u32 cons_slow(u32 a, u32 d) {                  // out of line — has function calls
  u32 i;
  if (freelist != FREE_END) { i = freelist; freelist = cells[i].cdr; }
  else {
    if (gc_enabled) { gc_a = a; gc_b = d; gc(); }
    if (freelist != FREE_END) { i = freelist; freelist = cells[i].cdr; }
    else { g_oom = 1; return ERR; }
  }
  cells[i].car = a; cells[i].cdr = d; return mkcons(i);
}

static inline u32 cons(u32 a, u32 d) {
  if (cell_top < MAX_CELLS) {                          // fast bump path
    u32 i = cell_top++;
    cells[i].car = a; cells[i].cdr = d;
    return mkcons(i);
  }
  return cons_slow(a, d);                              // can reach gc() — that's the barrier
}

At every cons callsite in the VM hot loop, the compiler now sees that the function could call cons_slow — which could mutate cell_top, write to the cells array, trigger garbage collection, modify the freelist, do anything. So the compiler has to preserve more state around every cons call. It pessimizes the surrounding hot loop.

Crucially, this tax is paid per callsite regardless of whether the slow path actually runs. On a benchmark where the bump path always succeeds and gc() never fires, the tax is still there in the compiled code. We can fire zero collections during a benchmark and still pay 1.8×.

How much pessimization happens depends on each engine's hot loop shape. bytecode_gc's switch dispatch has many arms that never touch cons (OP_LOADL, OP_JMP, the inlined arithmetic) — V8 can keep specializing those arms tightly. cek_gc's tight musttail chain is the case V8 specializes hardest, so it has the most to lose when forced to be conservative; it pays the highest amplification. lisp_gc's one-big-recursive-function shape is in between — less specialization opportunity to begin with, so less to lose.

5The H2 zero floor: lifting an idea from tinylisp

If the tax is "per cons callsite reaching a function call," it should disappear when no function call is reachable from cons. Robert van Engelen's tinylisp — "Lisp in 99 lines of C" — demonstrates exactly this pattern.

tinylisp's entire garbage collector is one line:

void gc() { sp = ord(env); }

It works because tinylisp's cell heap is a stack growing downward (cell[--sp]) and the global env head is always at the lowest used index. Every cons(a, b) produces a cell with index lower than min(a, b), so the set of cells reachable through env is exactly cell[sp..N]. gc() resets sp to the env head's index. Anything below sp is unreachable; future allocations overwrite it. O(1), perfectly precise, no tracing.

The catch is that this only works at boundaries where the only roots are the global env — in tinylisp, between top-level forms at the REPL prompt. It's not a general-purpose collector; closures created mid-eval that aren't bound to env can't survive past their creating top-level form. But for the wallisp benchmark suite, every program is one top-level form, so collection is never needed mid-bench. The cons hot path is what matters.

So we ported it. lisp_region.c's entire collector and its cons:

// cons — pure local mutation. No function call, no fast/slow split.
static u32 cons(u32 a, u32 d) {
  if (sp == 0) { g_oom = 1; return ERR; }
  sp--;
  cells[sp].car = a;
  cells[sp].cdr = d;
  return mkcons(sp);
}

// gc — the entire collector. Drop everything above the live env chain head.
static void gc(void) {
  g_numgc++;
  sp = considx(cdr(g_head));
}

The cons here is simpler than lisp.c's no-GC cons. There's no function call anywhere on either path; the compiler can treat cons as side-effect-free with respect to anything outside the cells array. Pre-registered prediction: lisp_region / lisp_big ≈ 1.0× within noise.

Measured: 0.94× on wasm, 0.99× on native. Region-drop is consistently faster than the no-GC tree-walker baseline. We predicted equivalence and got a small improvement.

The sub-1.0× result was actually flagged in the falsification log as the case to watch for. It implies the no-GC lisp.c baseline itself was paying a small cons-shape tax we hadn't isolated.

lisp.c's cons does if (cell_top >= MAX_CELLS) return ERR; cells[cell_top++] = ... — comparing a counter against a large constant. lisp_region.c's cons does if (sp == 0) ...; cells[--sp] = ... — comparing against zero, which compiles to a single instruction (i32.eqz in wasm, tbz on ARM). The comparison is one wasm op shorter, the loop is one wasm op shorter, and on V8 that compounds into a measurable 6% over the benchmark.

The H2 mechanism story sharpens: the "zero tax" pattern isn't "make cons match the no-GC baseline." It's "make cons the simplest possible mutation — no function call, comparison against zero, monotonic decrement — and the result lands below the no-GC baseline because even the baseline had headroom."

6Methodology: measure, don't guess

The project's working norm is "pre-register a prediction, then test it." Every hypothesis above was written down before the benchmark ran. The structure goes:

  1. State a hypothesis. E.g., "H4: GC widens bytecode's lead under a tight heap."
  2. Derive testable predictions. "bytecode_gc / cek_gc time ratio should be greater than bytecode / cek ratio."
  3. Specify falsification conditions. "If on any benchmark the ratio is below the no-GC ratio, that falsifies the prediction."
  4. Run the benchmark. Read the numbers.
  5. Update. If confirmed, the hypothesis stands. If refuted, write down which prediction broke and why — sometimes the mechanism story needs rephrasing, sometimes the whole hypothesis needs to die.

A surprising fraction of "obviously correct" interpreter-design intuitions get falsified by this process. Things we believed, then measured, then deleted from the working model:

  • The env-lookup hotspot. "Looking up variables in a cons-cell assoc-list is the bottleneck." We replaced it with an O(1) array. Speedup: about 5%. Not the bottleneck.
  • The switch-dispatch win. "Computed gotos / direct-threaded dispatch will beat the switch by 20%." Measured: 5%, then eaten by -Oz. Not the bottleneck.
  • CEK's tail-call advantage. "The tree-walker will blow the C stack on deep tail recursion." clang's -O2 made it unbounded for free. (Section 2 above.)
  • The GC tax is collection cost. "Most of the GC overhead is the mark/sweep work." Falsified: collection cost is negligible on benchmarks that fire 1–4 cycles. The dominant cost is the per-callsite optimization barrier — pessimization that's paid even when collection never runs. (Section 3 above.)
  • GC tax scales with allocation rate. "More allocations means more amplification." Falsified backwards: allocation-heavy benchmarks (nrev) show the least amplification, because both sides are already cons-dominated. Compute-bound benchmarks show the most.

The last two are the load-bearing ones; both came from H4. Each refutation required adding a new engine to the experiment (cek_gc, then lisp_gc, then lisp_region) to nail down the mechanism. The mechanism story now stands on four GC engines (three mark-sweep, one region-drop) and is sharper for it.

Why measurement-driven design matters more for interpreters

Interpreters live inside a compilation stack. Source C, clang's optimizer, wasm bytecode, V8's JIT — each layer transforms the code, and the layer above can erase or amplify what you did one layer down. Reasoning from the C source about runtime performance is reasoning across four substrates you don't see. Several of the surprises above involved one substrate "fixing" a problem at a different level than expected, or one substrate amplifying a cost at a different level than expected.

The native-vs-wasm decomposition turned out to be load-bearing. The same C code, compiled native vs compiled to wasm and run on V8, exposes which substrate is responsible for which effect. cek_gc pays a 1.28× tax native and 1.83× tax wasm; the gap between those is V8's amplification of the per-callsite barrier on CEK's musttail chain. Without the native baseline, we couldn't have decomposed the cost into "compiler-level" and "JIT-level" parts.

The workflow that emerged from this project:

  • Build a native binary of each engine for direct C measurement.
  • Run the same benchmark on wasm via Node/V8.
  • The ratio (wasm time) / (native time) is the substrate amplification factor.
  • The ratio (GC engine) / (no-GC engine) is the GC tax.
  • Decompose: tax has a native floor (compiler-level) and a wasm amplification (JIT-level). Both are real, neither is the whole story.

That's the actual transferable result of this project, more than any single engine. Four substrates means four places for performance to come from or vanish into; you can only find out by measuring all four.

7The ceiling: hand-written JS and C

The methodology above describes a four-substrate ladder — C source, clang's optimizer, wasm, V8's JIT — and the workflow for decomposing performance across it. There's a natural follow-up: what would these algorithms cost with no interpreter at all? Hand-write the same five benchmarks in JavaScript and in C, run them on the same V8 and the same machine, and we get two reference points sitting above all eight engines — the upper bound on what these programs can cost on this hardware.

The hand-written versions are in baselines/bench.{js,c}. Same algorithms shape-for-shape: fib/tak/ack stay recursive; nrev+sum uses cons-cell shaped linked lists (switching to arrays would change the asymptotic memory pattern and make the comparison dishonest); tailsum in JS is a while-loop because V8 has no proper TCE.

Against bytecode_gc (the finalist engine) on the same hardware, best-of-25:

Two surprises in the table.

JS beats C on nrev+sum. The only row where the gap from the interpreter to JS is wider than the gap to C. The reason is allocator shape: one iteration does about 11,000 cons allocations, each a short-lived three-word object. V8's young-generation bump allocator handles those at near-zero amortized cost; glibc's malloc pays book-keeping per call. This is a well-known V8 advantage and our benchmark hits it head-on. On the compute-only rows (no allocation), C wins over JS by the expected 2–3×.

tailsum(30000) in C clocks at 0.000 ms. clang at -O2 sees through the tail recursion and rewrites the algorithm as n*(n+1)/2 — closed-form arithmetic. The reported "time" is below the timer's resolution because the work isn't being done at runtime; it was done at compile time. The Lisp engines obviously can't do this — the program isn't visible to clang. This is the deep reason an interpreter, however well-written, has a ceiling it can't reach: it can't see the whole program.

So the full substrate ladder for fib(24) on this machine, fastest at the top:

  • C, -O2 native — ~0.13 ms
  • JS-on-V8 (V8 JITs the source) — ~0.33 ms
  • bytecode_gc, wasm-on-V8 (V8 JITs the interpreter, which interprets the program) — ~7.3 ms
  • cek_gc, wasm-on-V8 (worst engine, same shape, no inlined prims) — ~57 ms

The engine work in this project moves us from 57 ms to 7.3 ms — about 8×, on the interpreter side of a fixed substrate. From 7.3 ms to 0.13 ms is the remaining ~55×, and it lives in compiler/JIT territory we don't control. The headroom is real, and it's worth knowing where it actually sits.

8A note on -Oz

The whole project is built with clang -O2. -Oz appears once in the falsifications above — "the thing that ate the switch-dispatch speedup." We never quantified its size effect, which turns out to be the more interesting story.

Rebuilt every engine at both flags, same toolchain, same link settings:

Three things to notice.

-Oz halves every artifact. 47–61% smaller, average 55%. -Oz aggressively prefers code-size over hot-path inlining; the wasm modules lose duplicated sequences and become friendlier to V8's tiered JIT (less code to baseline-compile, smaller hot paths to specialize, better instruction-cache behavior).

On most engines, V8 prefers the smaller code. The tree-walker family and — importantly — bytecode_gc (the finalist) run faster at -Oz. bytecode_gc lands at −11% time and −53% bytes simultaneously. Strict win.

CEK regresses catastrophically. cek is 2.6× slower at -Oz; cek_gc is 1.9× slower. The mechanism is consistent with the prior native-vs-wasm finding: CEK's tight musttail chain is the shape V8 specializes hardest, and -Oz emits a coarser code shape (fewer inlined dispatch arms) that V8's tier-up pattern-matchers don't recognize as the hot musttail loop. What makes CEK fast on V8 is exactly what -Oz declines to spend code-size on.

So why does build.sh still default to -O2? Because a single flag that doesn't make any engine worse is the safe shared setting for a comparison study. An opinionated build script would use -Oz for everything except cek* and ship the finalist at roughly half the byte count. The deeper point: clang's -O2/-Oz choice interacts with V8 in a way that isn't predictable from either layer alone — -Oz is "smaller and slower" on most native targets, and on V8 it's "smaller and usually equal or faster, but CEK 2× slower." Four substrates, four places for the answer to flip.

Closing — eight engines, four hypotheses, three falsifications

The full project ran four pre-registered hypothesis arcs:

  • H1: CEK is needed for unbounded tail recursion. Refuted. clang TRE made the tree-walker unbounded for free. Later verified with lisp_trampoline.c (1.005× wasm vs lisp.c, 2 bytes apart in module size).
  • H2: GC overhead is the cost of mark/sweep collection. Refuted. The dominant cost is the per-cons-callsite optimization barrier, paid even when collection never runs. Region-drop demonstrates the zero floor.
  • H3: Mark cost vs sweep cost composes a particular way. Not yet measured.
  • H4: GC widens bytecode's speed lead. Partially confirmed; mechanism refuted. The widening is real (4 of 5 wasm benchmarks) but driven by H2's per-callsite barrier amplified by engine shape, not by collection frequency. Three GC engines together show V8's amplification scales with each engine's JIT-specializability: bytecode_gc ~1.0×, lisp_gc ~1.1×, cek_gc ~1.4×.
