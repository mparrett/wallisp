Partial evaluation

When you stop interpreting

Two ways to remove the eval loop from a tiny Lisp. Both predicted small wins. Both delivered 50–65×.

This page is a follow-on to the main wallisp study, which built the same Lisp eight ways and compared engines on the same benchmarks. Bytecode beat the tree-walker by ~2.5×. A four-rung optimization ladder inside the bytecode line (bc_orig → bc_base → bc_inline → bc_super) found one more 1.6× by hoisting primitive dispatch from the VM to the compiler. The pattern: each rung does more at compile time, less at runtime.

What if the limit of that ladder is "do all of it at compile time"? Partial evaluation says: specialize the interpreter against a particular source program, and the result is the source program without the interpreter. That's the Futamura projection — old idea, well-known on paper. We hadn't measured it on our engines.

We tried two flavors: H9, leaving Lisp entirely (Lisp→C residual, compile to wasm, run); and H10, staying in Lisp with an opt-in $ comptime marker. Both predictions were 3–5×. Both came in 10–20× larger than predicted. The misses are the interesting part.

1Setup — when the eval loop is the cost

The bytecode VM’s hot loop dispatches through a br_table over 14 opcodes. V8 specializes each arm hard — we’d already verified this in the wasm-dispatch study (see docs/project_notes/wasm_dispatch.md). The per-opcode cost is genuinely small. So removing dispatch shouldn’t buy much: bytecode is already 2.5× faster than the tree-walker, and that’s the dispatch-removal layer.

The optimization ladder confirmed the diminishing-returns story. Each rung trades generality for speed:

rungwhat it addsvs base
bc_basebaseline (apply_prim, args consed)1.0×
bc_inlineVM checks is_prim at runtime, inlines 2-arg arith1.2×
bc_supercompiler emits dedicated OP_PADD… superinstructions1.6×

So the natural prediction was: removing the eval loop entirely should buy maybe one more 1.6×-shaped step. Maybe 3–5× in total, since you also remove the call-frame and operand-stack accounting. We pre-registered that.

The prediction was 10–20× off.

2H9 — leaving Lisp entirely

The Futamura projection #1, by hand: write a C function that does what (fib 24) would do, with the same 30-bit tagged-fixnum representation as engines/lisp.c. The interpreter loop is gone; the value rep is preserved. Compile to wasm with the same flags as the engines, run through the same eval_source ABI.

// residual_fib_tagged: PE residual, dispatch removed, value rep preserved.
#define mkfix(n)   ((u32)(((i32)(n)) << 2))
#define fixval(v)  (((i32)(v)) >> 2)

static u32 fib(u32 n) {
  if (fixval(n) < 2) return n;
  u32 a = fib(mkfix(fixval(n) - 1));
  u32 b = fib(mkfix(fixval(n) - 2));
  return mkfix(fixval(a) + fixval(b));
}

Pre-registered prediction (stated mid-session): bytecode_gc / residual ratio of 3–5×. The dispatch-removal logic that produced bytecode’s 2.5× lead over the tree-walker, applied one more time.

residual: taggedresidual_fib_gen.wasm0.150 ms128.7×

Predicted 3–5× over bytecode_gc. Measured 50.9×. The pre-registered model was 10–20× too small.

The naive model treated the cost as per-opcode dispatch (which V8 specializes). The actual cost is the count of opcodes per recursive call — for fib(24)’s ~150K recursive calls, each does LOADL × params, LOADG for the recursive self-reference, CALL with arg-list construction, RET with frame teardown: ~50 dispatched opcodes per call, ~7.5M total. The residual collapses every call to one C function call. V8 inlines and specializes a recursive C function much more aggressively than it inlines a run loop that contains a recursive dispatch.

This amplifies the H8 finding from the main study: V8 specializes the bytecode VM’s br_table per-arm, which is real but bounded. The remaining cost is the per-call accounting that no per-arm specialization removes. Only structural collapse — PE — does.

The specializer

The hand-written residual is the proof of concept. The thing that makes it Futamura #1 is a specializer: a host-side program that reads any Lisp source and emits the residual mechanically. prototype/futamura/specialize.c is ~280 lines that walks the AST — if a node is comptime-known (if head, + head, parameter ref), it picks the branch; if a node is runtime (a function call with runtime args), it emits the C call. The pattern is “eval-or-emit”: every AST node either folds to a value or emits residual code, never both. (Background on the technique generally: Wikipedia: partial evaluation.)

Input fib.lisp

(define (fib n)
  (if (< n 2)
      n
      (+ (fib (- n 1))
         (fib (- n 2)))))

Generated residual (excerpt)

static u32 fib(u32 n){
  return (fixval(mkfix(fixval(n) < fixval(mkfix(2))))
          ? n
          : mkfix(fixval(fib(mkfix(fixval(n) - fixval(mkfix(1)))))
                + fixval(fib(mkfix(fixval(n) - fixval(mkfix(2)))))));
}

The generator’s output is verbose. fixval(mkfix(x)) is just x; the generator doesn’t simplify, it just emits. clang’s -O2 folds the redundancy. The wasm produced from this is byte-identical to the hand-written tagged residual — verified via cmp on both fib and tak. The specializer hits the ceiling for tagged-rep PE without needing its own simplification pass.

Cross-check on tak(18,12,6): bytecode_gc / residual = 65.4×. Same shape as fib, different program. The 50–65× ratio is robust, not fib-specific.

Bonus: size collapses too

PE removes work and removes code in the same move. The fib residual is 820 bytes of wasm; bytecode_gc.wasm is 26 KB. The cost paid for both wins is generality — the residual runs only fib.

Same mechanism behind both wins: collapsing the dispatch loop drops the count of instructions executed (speed) and the amount of code that has to exist (size). One move, two numbers.

3H10 — staying in Lisp with $

H9 is the most aggressive thing PE can do: leave the source language entirely. The opposite end of the spectrum is "do the same eval-or-emit walk, but emit Lisp." The engines stay the runtime; PE just hands them a smaller program with some subexpressions pre-folded.

The honest blocker is that finding foldable subexpressions automatically is hard. Binding-time analysis — the standard PE technique — is a lot of code and prone to either missing folds or being conservative. Our shortcut: let the programmer mark them. A $ prefix on a form means "fold this at preprocess time." If it can’t be folded, preproc errors out.

(define (sum-to n)
  (if (= n 0) 0 (+ n (sum-to (- n 1)))))

(define (work x)
  (+ x $(sum-to 100)))      ; $-marked: fold (sum-to 100) at preprocess time

(define (loop n acc)
  (if (= n 0) acc
      (loop (- n 1) (+ acc (work n)))))

(loop 1000 0)

After preproc:

(define (sum-to n) (if (= n 0) 0 (+ n (sum-to (- n 1)))))
(define (work x) (+ x 5050))       ; folded
(define (loop n acc) (if (= n 0) acc (loop (- n 1) (+ acc (work n)))))
(loop 1000 0)

(sum-to 100) is evaluated once at preprocess time, by a small in-host evaluator with a 200K-reduction fuel cap, and the result 5050 is substituted into the AST. The engines never see the $; they receive plain wallisp source. Both versions of the program produce 5550500.

engineunmarkedfoldedspeedup
tree-walker (lisp_big)18.52 ms0.269 ms68.9×
bytecode_gc4.94 ms0.083 ms59.9×

This 60× is not a general PE claim. It’s the reuse factor of this benchmark. The demo deliberately calls (work x) 1000 times in a loop; each call would have done a 100-step sum-to recursion. Used once at top level, $(sum-to 100) saves one sum-to’s worth of runtime — invisible.

The honest expression of the win is runtime_uses_of_($-fold_result) × cost_per_evaluation. The 60× is a knob you turn by structuring the program around reuse. H9’s 50×, by contrast, is structural: collapsing the eval loop benefits every program because every program runs through it.

Hard-fail by design

$E is a promise: it folds, period. If folding fails — unbound symbol, unknown function, fuel exhausted, unsupported construct — preproc prints a diagnostic and exits non-zero. No soft fall-through to runtime.

$ echo '(define (f x) $(+ x 1))' | preproc
preproc: $-fold: unbound symbol `x`
exit: 1

$ echo '(define (f x) $(no-such-fn 1 2))' | preproc
preproc: $-fold: unknown function `no-such-fn`
exit: 1

This was a design call. The alternative — soft-fail, where an un-foldable $E silently strips the marker and runs at runtime — is friendlier but introduces a silent performance cliff. The user thinks they’re getting compile-time work; they’re not. Hard-fail catches the most common comptime bug (passing a runtime value into a comptime slot), makes $E mean something definite, and is actually simpler to implement: "try to fold, error on failure" rather than "distinguish graceful unfold from a real bug."

Convergent signal: barrier() in the comptime specializer drop that prompted this whole thread, Zig’s comptime keyword, and C++ constexpr all chose hard-fail. Different languages, same call.

Dev affordance: --trace

Treated as a debug utility, the preprocessor benefits from a trace mode. preproc --trace writes one line per fold to stderr; stdout is unchanged.

$ preproc --trace < comptime_demo.lisp > folded.lisp
$(sum-to 100) => 5050

Combined with diff folded.lisp original.lisp, this tells you both what changed in the source and why. A plain source diff doesn’t carry the “why.”

4Same theory, two surfaces

H9 and H10 are both Futamura projection #1 — specialize the interpreter against a source program — but they make opposite choices about where the residual lands.

axisH9 (Lisp→C→wasm)H10 (Lisp→Lisp)
targetC residual, compile to wasmfolded Lisp source
substrateleaves Lisp entirelystays on existing engines
toolchain addedspec + clang + wasm-ldspec, one pipe
win sourceeval loop removedpartial subexpression folded
win generalitystructural, every programconditional on reuse
win magnitude50–65×60–69× (demo-shaped)
engineering~280 LOC + new toolchain~340 LOC, no toolchain
programmer effortnonemark sites with $

These are complementary, not competitive. H9 is "your whole program is comptime-known — ship a residual binary." H10 is "parts of your program are comptime-known — fold them, run the rest normally on the engines you already have."

The magnitudes happen to be in the same neighborhood because both demos are constructed to make PE look good. The mechanisms are different:

  • H9’s 50× is the per-call opcode count of the bytecode VM, multiplied by the call count of the program. Every program pays this. Removing the eval loop benefits every program equally.
  • H10’s 60× is the reuse factor of the folded subexpression. A program that doesn’t reuse the result of any $E sees nothing. A program that reuses it in a hot loop sees linear-in-iterations savings.

One way to think about the relationship: H10 is "what wins fit inside Lisp." H9 is "what wins fit outside Lisp." On the demos we picked, both pay around 60×. On a program where the whole computation could be folded, H10 would be infinite (the program is the fold) and H9 would be irrelevant (nothing left to run). On a program where nothing is comptime-knowable, H10 would be 1× (nothing to fold) and H9 would still be 50× (the eval loop is gone regardless). The methods don’t overlap; they cover different parts of the program-space.

5What’s open

Three honest threads were left untouched on this pass.

Cons cells and closures. The H9 specializer and the H10 preproc both stop at fixnum-only programs — no cons, no closures-over-locals, no let or cond in the folded portion. Extending the folder to handle cons/car/cdr is mechanical and unlocks lookup-table programs. Closures need a representation in the host evaluator that tracks free-variable capture — bigger, and worth measuring whether the marginal complexity pays off on the metacircular evaluator (where most of the cost is closure dispatch).

Cross-runtime measurement. The 50–65× numbers are wasm-on-V8. Bun (JavaScriptCore) and the native build (no JIT) are easy follow-ons given existing harness infrastructure. JSC inlines wasm differently from V8; the gap may compress or widen. The main study has cross-runtime parity for the engines — the residuals should run through the same harness without changes.

Composition. Run H10 first to fold $-marks, then H9 to specialize the folded program to C. Untried. Plausibly noisy — H9 already removes the cost of the engine; H10 only matters if the engine is going to run the program. Putting them in series may give H9 a smaller program to chew on, which clang then optimizes better, but the speedup is probably indistinguishable from H9 alone on programs where H9 already wins big.
